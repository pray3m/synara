Implementation architecture · Plan 006

Synara is the harness.

A control-plane plan that makes every provider recognize Synara first, routes Synara-resource actions through Synara MCP, and prevents detached agents or retries from multiplying threads.

The key decision

Do not rely on a stronger prompt alone. Combine provider-native host instructions, truthful capability discovery, and server-enforced turn-scoped idempotency.

Architecture Target flow Milestones Tool surface PR #2829 Risks Proof

Three layers, one invariant

The model should understand the host, but the server must remain correct when the model misunderstands, retries, or continues in a detached background task.

1

Model awareness

A single versioned Synara host policy is delivered through Codex developer instructions, Claude system append, and the safest available provider-specific channel everywhere else.

“I am running inside Synara. Synara owns threads and automations.”

2

Capability truth

synara_context and synara_capabilities expose canonical providers, models, reasoning options, and honest limitations before a destructive action.

No guessed gpt-5.6-terra-low slug and no silent fallback.

3

Server authority

Writes require a live provider session and active parent turn. One turn commits one exact creation batch; deterministic ids make retries return the same result.

Detached background agents cannot create thread after thread.

The target request flow

The happy path is one batch and one bounded wait. The historical detached retry path is rejected before any thread or worktree side effect.

Implementation sequence

The order keeps diagnosis, policy, authorization, and persistence separate so each layer can be verified before coordination behavior changes.

01CharacterizeLock the six-thread incident into failing tests.
02RecognizeCentral host policy through every provider's strongest channel.
03AuthorizeSession-scoped credentials and active-turn write checks.
04ValidateCanonical provider, model, and option resolver.
05ReserveOne exact idempotent creation plan per caller turn.
06CoordinateBounded wait, final synthesis, recovery replay.

A tool surface models can reason about

Plural work becomes a first-class operation. Failures are data to report, not permission to invent replacement workers.

synara_context

Who and where am I?

Returns the Synara harness identity, caller thread/turn/provider, and available host capabilities.

synara_capabilities

What can Synara actually run?

Returns the canonical model catalog and supported reasoning options used by server validation.

synara_create_threads

Create the exact batch once

Validates 1–20 entries atomically, reserves one plan, and replays the same result for the same request.

synara_wait_for_threads

Wait without side effects

Pins each selected run and returns ordered status, summary, and errors with a short rolling timeout.

PR #2829: inspiration, not architecture

The external draft contains useful narrow patterns, but its 710-file V2 migration is neither necessary nor safe to import wholesale.

Adopt selectively

  • Provider-session MCP invocation scope
  • Capability and canonical target validation
  • Stable command/thread/message ids
  • Batch creation and explicit wait
  • Creation provenance and tool annotations

Inspect the upstream orchestration prototype.

Reject or change

  • No full orchestration V2 rewrite
  • No predecessor product names inside Synara
  • No random fallback when idempotency key is omitted
  • No ten-minute default MCP wait
  • No Claude MCP-only allowlist that hides native tools

Risks that deserve reviewer attention

Risk Likelihood Mitigation
Detached process races turn completion High Resolve current active turn at write time; terminal transition wins.
Duplicate worktree or partial thread batch High Reserve before side effects, deterministic ids, ownership-aware compensation, startup recovery.
Dynamic model catalog temporarily unavailable Medium Allow configured default only; fail closed for unverified custom targets.
Provider host text competes with native prompts Medium One concise versioned policy, delivered exactly once through the strongest channel.
Pooled provider leaks another thread's token High Keep MCP capability false unless per-session binding exists; never put a thread token in shared config.
Tool annotations create false confidence Low Treat annotations as advisory; tests assert backend idempotency and authorization.

The acceptance test

Replay the original instruction and deliberately attempt the old failure path. Start two workers—Codex GPT 5.6 Terra with low reasoning and Claude Sonnet 5—ask both what the repository is about, wait for both, and synthesize. Then simulate a detached native subagent retrying invalid and alternative targets. The final database must contain exactly two operation-owned threads, no OpenCode fallback, canonical Codex options, and one parent synthesis.
